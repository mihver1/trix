# Task: XEP-0308 Edit And XEP-0424 Retraction

You are the next coding agent working in the Trix repo. Add safe editing and
retraction for own text messages with OMEMO-preserving semantics.

## Current Context

Relevant files:

- `docs/security.md`
- `docs/xmpp-migration/protocol-feature-map.md`
- `apple/README.md`
- `apple/Sources/Shared/Models/TrixModels.swift`
- `apple/Sources/Shared/Services/TrixServiceProtocols.swift`
- `apple/Sources/Shared/Services/XMPPMartinService.swift`
- `apple/Sources/Shared/Services/MockTrixService.swift`
- `apple/Sources/Shared/ViewModels/TimelineViewModel.swift`
- `apple/Sources/Shared/Views/TrixTimelineView.swift`
- `apple/Sources/Shared/App/XMPPLiveSmokeRunner.swift`

Current timeline items store `id`, `sender`, encrypted body-derived text,
attachment metadata, delivery state, and reactions. There is no edit history or
tombstone state.

## Goal

Users can edit or retract their own text messages. For the first XEP-0308
slice, enforce "last editable own text message" semantics unless you explicitly
validate a broader client-compatible convention. Edit replacement text is
OMEMO-encrypted; edit/retraction target metadata remains XMPP-visible and is
documented as metadata.

## Non-Goals

- Do not edit messages from other users.
- Do not edit attachments, stickers, voice messages, reactions, or retracted
  messages in the first slice.
- Do not claim retraction erases MAM history or deletes copies from other
  devices. Render it as a tombstone state.
- Do not add plaintext replacement text outside OMEMO.

## Implementation Plan

1. Confirm stable message identifiers:
   - DMs may use the client-generated message id if MAM preserves it.
   - MUCs often need server stanza ids or origin-id mapping. If current group
     ids are not stable across devices/restart, add that mapping first.
2. Extend `TrixTimelineItem` with edit metadata:
   - `isEdited`;
   - optional `editedAt`;
   - optional `isRetracted`;
   - optional `retractedAt`;
   - optional `replacementOfMessageID` or event history if needed.
   Keep cache decoding backward compatible.
3. Add service APIs:
   - `editText(messageID:roomID:newText:session:)`;
   - `retractMessage(messageID:roomID:session:)`.
4. Implement XEP-0308:
   - Send an encrypted replacement message body.
   - Add the XEP-0308 `<replace/>` target metadata.
   - Enforce "last message correction" scope first; do not silently allow
     arbitrary historical edits unless interoperability has been validated.
   - Apply replacement locally only if sender matches the original item owner.
5. Implement XEP-0424:
   - Send a retraction message for the target id.
   - Replace local body with a neutral tombstone such as "Message deleted".
   - Preserve enough metadata for timeline ordering and audit.
6. Parse incoming edit/retraction events from direct messages, carbons, MAM, and
   group messages.
7. Add UI:
   - own-message context menu edit and delete actions;
   - edit composer state;
   - edited marker;
   - retracted tombstone.
8. Add live smokes:
   - `dm-edit-retract`;
   - `group-edit-retract` only after group id stability is proven.
9. Update `docs/security.md` with visible metadata notes for edit/retract target
   ids and timestamps.

## Acceptance Criteria

- Own DM text message can be edited and the peer sees the replacement.
- Own DM text message can be retracted and the peer sees a tombstone.
- Attempting to edit/retract another sender's message is blocked locally and
  ignored on receive.
- Replacement text is encrypted through the same OMEMO path as normal text.
- Existing reaction and attachment messages are not broken.
- Restart/MAM reload preserves edited/retracted state.
