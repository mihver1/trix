# Task: XEP-0372 Mentions

You are the next coding agent working in the Trix repo. Add @mentions using
XEP-0372 references while preserving OMEMO privacy expectations.

Current group member data comes from MUC occupants, affiliation queries, and the
Keychain known-member cache. Push handling is wake-only and local notifications
are generic.

## Goal

Users can mention known room members with `@name`. The outgoing stanza carries
XEP-0372 reference metadata, the receiving client highlights mentions, and local
inactive notifications can use a stronger generic mention wording after local
sync/decrypt.

## Non-Goals

- Do not put message text in APNs payloads.
- Do not include decrypted mention context in server-visible metadata.
- Do not add `@everyone` or broadcast mentions in the first slice.
- Do not mention users outside the known DM peer or joined MUC member set.

## Implementation Plan

1. Add mention parsing and composer UX:
   - detect `@` in the composer;
   - show known room members from directory/group cache;
   - insert a display token that maps to a real JID.
2. Add mention metadata to the send path:
   - attach XEP-0372 `<reference/>` elements with mention type and `xmpp:` URI;
   - calculate begin/end offsets according to the XEP and validate Unicode
     handling carefully.
3. Keep the actual message body OMEMO-encrypted. The visible reference metadata
   should identify the mentioned JID/range but must not expose decrypted context.
4. Parse incoming XEP-0372 references and add mention metadata to
   `TrixTimelineItem`.
5. Render mention highlighting in timeline bubbles.
6. Extend inactive local notification logic:
   - APNs remains wake-only;
   - after local sync/decrypt, a message mentioning the current user may produce
     generic copy such as "You were mentioned";
   - never include the message body, room title if sensitive, attachment
     filename, or decrypted context.
7. Add live smoke `group-mention`:
   - three accounts in a private MUC;
   - owner sends an encrypted message mentioning peer;
   - peer receives/decrypts and sees mention metadata;
   - status output prints only IDs/counts/booleans.
8. Update `docs/security.md` to document that mention target JIDs and reference
   offsets are server-visible metadata.

## Acceptance Criteria

- Mention picker works from the current room member source.
- Mention metadata is sent and parsed in DMs or groups, with group coverage
  preferred.
- Mentioned messages are highlighted locally.
- Mention-triggered local notifications remain generic and plaintext-free.
- Unknown users cannot be injected as mentions by raw text alone.
- Live smoke validates at least one group mention without printing message text.
